## 分析：当前问题与可应用的方案

### 当前问题的本质

你现在的代码已经走了一半正确的路：

```42:44:server/src/prompts/scene-gen.ts
function pickHouseType(): (typeof HOUSE_TYPES)[number] {
  return HOUSE_TYPES[Math.floor(Math.random() * HOUSE_TYPES.length)]!;
}
```

`pickHouseType()` 用 `Math.random()` 选出了一个随机的屋子类型（比如"织布人家"），然后把这个类型名称直接注入到 prompt 里：

```70:71:server/src/prompts/scene-gen.ts
- **本次屋子类型：${house.name}**（${house.traits}）。你必须使用这个屋子类型，场景描述、区域划分、物品设计都必须与之吻合。
```

**问题在于**：屋子类型虽然随机了，但屋子类型只是一个"分类标签"，LLM 面对同一个标签（比如"织布人家"），每次联想出的具体物品、场景描述、区域划分还是会趋同——因为上文（屋子类型名）是相同的，LLM 的下文输出就会高度相似。

---

### 文档方法的核心原理与适配方案

文档的核心是：**用代码生成随机字符串/词汇，注入 prompt，让 LLM 基于这个随机上文"联想"出不同的具体内容**。

针对你这个项目，可以在**两个层面**应用这个方法：

---

#### 方案一：给物品生成注入随机"联想种子词"（最直接、收益最大）

当前物品设计完全依赖 LLM 自由发挥，导致每次"织布人家"都会生成织机、纱线、染缸这几样老面孔。

**做法**：在 prompt 里额外注入一组用代码随机生成的"随机汉字拼音首字母串"（类似文档中的 `xcxzv`），然后让 LLM 先将这些字母联想为具体的中文词汇，再将这些词汇作为物品创意的灵感来源融入场景设计中。

示例 prompt 追加内容：

```
## 本次创意种子（必须参考）
随机字符串：${randomSeed}  
请先将每个字母联想为一个与当前屋子环境相关的中文名词，再将这些词汇作为灵感，融入物品设计中（不必强行照搬，但要有所体现）。
```

这样同样是"织布人家"，一次可能联想出"织机旁的碎陶片"，另一次可能联想出"染缸边的旧皮鞋印"，内容就真正随机了。

---

#### 方案二：给区域布局注入随机"方位组合种子"

当前区域（zones）也容易固定化（每次织布人家都是窗边+屋中央+角落三区）。

**做法**：用代码随机生成 2-4 个抽象方位词或空间形容词组合，作为本次区域划分的灵感约束注入 prompt：

```
## 本次区域灵感词（必须参考）
${randomZoneWords}  
本次场景的区域划分请从这些词汇中获取灵感，设计出与之相关的空间结构。
```

---

#### 方案三：给场景氛围注入随机"感官细节种子"

场景的 `description` 总是"阴暗、潮湿、昏迷醒来"的固定开头。

**做法**：随机生成几个感官词（嗅觉/听觉/触觉），注入 prompt 要求 LLM 必须将这些感官细节融入场景描述中：

```
## 本次感官细节要求（必须融入description）
${randomSensoryWords}
```

### 最推荐的实施优先级

| 优先级 | 方案 | 解决的问题 | 实现复杂度 |
|--------|------|-----------|-----------|
| ⭐⭐⭐ 最高 | 方案一：物品联想种子 | 物品内容高度重复 | 低，加一个随机字母串生成函数 |
| ⭐⭐ 中 | 方案二：区域布局种子 | 区域结构固定 | 低 |

### 代码层面的随机串生成方式

```typescript
function generateRandomSeed(): string {
  const letters = 'ABCDEFGHIJKLMNOPQRSTUVWXYZ';
  const digits = '0123456789';
  const c1 = letters[Math.floor(Math.random() * 26)]!;
  const c2 = digits[Math.floor(Math.random() * 10)]!;
  const c3 = letters[Math.floor(Math.random() * 26)]!;
  const c4 = digits[Math.floor(Math.random() * 10)]!;
  const c5 = letters[Math.floor(Math.random() * 26)]!;
  return `${c1}${c2}${c3}${c4}${c5}`;
}
```

这样就把"代码随机"完全限制在生成这 5 个字符上，后续所有具体内容（住所细节、屋主身份、物品特征等）全部交给 LLM 自由联想，既保证了真正的随机性，也充分利用了 LLM 的语言创造力，不再被 `HOUSE_TYPES` 数组的 7 个固定选项锁死。

### 代码生成函数

```typescript
function generateSceneSeed(): string {
  const letters = 'ABCDEFGHIJKLMNOPQRSTUVWXYZ';
  const digits = '0123456789';
  const pick = (pool: string) => pool[Math.floor(Math.random() * pool.length)]!;
  return `${pick(letters)}${pick(letters)}${pick(digits)}${pick(digits)}${pick(letters)}`;
}
```

格式：`字母字母数字数字字母`，例如 `WD47T`，对应：废庙/守望兼住 + 山野谋生者 + 暂住借住 + 清贫寥落 + 修补角。

## 推荐：直接加入同一个 LLM 请求

**理由如下：**

**随机串的本质是"上文锚点"，不是需要推理的任务。** 文档里说的核心原理是：给 LLM 提供不同的上文，让它联想出不同的下文。随机串的作用是改变 prompt 的输入，不是让 LLM 去"决策"房屋类型——LLM 只是按规则做字符映射，这个映射动作越简单越好，不值得单独开一个请求。

**两请求方案的问题：**
- 第一个请求让 LLM 把 `WD47T` 映射成 "废庙+猎户+暂住+清贫+修补角"，这个结果是一段结构化文本
- 第二个请求再拿这段文本生成场景
- 但第一步本质上只是查表，LLM 做这件事毫无优势，代码直接做映射更快更稳定，而且还多了一次延迟和 token 消耗

**同一请求的优势：**
- 随机串注入 prompt 后，LLM 在生成场景的同时完成映射，两件事在同一个推理过程中自然融合，生成的场景会更连贯一致
- 如果分两步，第一步的映射结果是"离散的标签"，第二步再把标签转化成场景，等于做了两次"标签→内容"的转换，反而更容易让结果标签化、公式化

---

**结论：** 随机串直接写进 `scene-gen.ts` 的 `buildMessages` 里，用代码生成种子，注入到同一个 user prompt 里。只有当你需要让"房屋决定"这件事本身有复杂的业务逻辑（比如根据游戏进度排除某些类型）时，才值得单独抽出来。现阶段完全不需要。
