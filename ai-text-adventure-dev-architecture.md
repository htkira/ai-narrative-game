# Web 原型开发文档（V1）

## 1. 文档目标

本文件用于指导 AI 进行 **Web 前端原型的子任务化、模块化开发**。
目标不是做完整产品，而是做出一个 **本地可运行、可完整游玩第一世第一幕的前端原型**。

本文件解决四类问题：

1. 明确这一版原型到底做什么、不做什么
2. 明确前端整体架构和模块边界
3. 明确交互事件流
4. 明确前端使用的接口数据模型字段，先用 Mock 数据实现，未来再平滑替换成真实 API

---

## 2. 本版开发结论

### 2.1 当前版本的最终定位

这是一个 **单页 Web 原型**，用于本地测试第一世第一幕的完整流程。

### 2.2 当前版本的技术边界

* 不接后端
* 不接真实 Dify
* 不接真实图片生成
* 不做存档
* 不做账号
* 不做多幕系统
* 不做复杂动画
* 不做真正的推理面板组合功能

### 2.3 当前版本必须实现的闭环

* 开始界面
* 开场预设演出
* 主游戏界面
* 探索阶段
* 获得残缺念珠
* 使用残缺念珠查看残念
* 线索收集完成
* 触发 NPC 辩论
* 追问 / 出示物证
* 驳倒主要观点
* NPC 说出白姑去向
* 本幕结束

### 2.4 当前版本的内容来源

当前所有动态内容均由 **Mock 数据提供器**提供，不依赖真实 API。

---

## 3. 与游戏设计的对应关系

本原型必须忠实承接以下设计原则：

* 第一幕是“探索获得局部真相，再通过辩论击破错误叙事”的开场关卡。
* 场景与物品可动态变化，但必须服从同一底层事件链。
* 残缺念珠是本幕固定出现的关键物件，必须被玩家发现，并承担“看到一念残影”的机制升级作用。
* 辩论不是让玩家复述真相，而是让玩家用物证驳倒 NPC 观点。
* 一旦进入辩论，不应要求玩家退回探索重新找线索。

---

# 4. 本版原型的明确答案

## 4.1 应用形态

本版采用 **单页应用（SPA）**。
不做多页面路由切换，所有界面都在一个应用内根据 `phase` 切换显示。

## 4.2 状态权威

本版由 **前端本地状态**作为唯一状态权威。
Mock 数据只提供内容，不决定主流程。

## 4.3 AI 替代方案

本版不接真实 AI，使用 `MockContentProvider` 提供所有内容。
未来替换为 `ApiContentProvider` 时，前端 UI 和状态结构不应重写。

## 4.4 场景图片策略

本版使用 **静态占位图**。
图片区域必须预留，但不依赖真实生成。

## 4.5 调试面板

本版必须带一个 **简易 Debug Panel**，用于本地测试。

## 4.6 推理面板

本版不实现正式推理面板。
只保留“线索计数”和“已发现物证列表”的轻量表现。

## 4.7 辩论轮次

本版辩论按 **3~4 个主要观点**进行。
只要大部分关键观点被驳倒，即可收束。

# 6. 前端整体架构

## 6.1 总体结构

前端原型分为四层：

### A. App 壳层

负责整体阶段切换与页面布局。

### B. 游戏状态层

负责维护第一幕全部真状态。

### C. 内容提供层

当前为 Mock 数据提供器，未来可替换为 API 提供器。

### D. 界面模块层

负责渲染各个界面与交互区域。

---

## 6.2 推荐模块划分

### 1. `AppShell`

作用：

* 管理全局布局
* 根据 `phase` 渲染不同界面
* 承载 Debug Panel

### 2. `StartScreen`

作用：

* 显示标题
* 开始游戏按钮

### 3. `OpeningScene`

作用：

* 播放开场预设演出
* 结束后进入主游戏界面

### 4. `MainGameScreen`

作用：

* 探索阶段主界面容器
* 承载场景图、场景描述、区域列表、物品列表、右下角残缺念珠、左下角线索数

### 5. `SceneImagePanel`

作用：

* 显示当前场景占位图

### 6. `SceneTextPanel`

作用：

* 显示场景标题与场景描述

### 7. `ZoneSelector`

作用：

* 显示当前可探索区域
* 处理区域切换

### 8. `ItemGrid`

作用：

* 显示当前区域下的物品图标

### 9. `ItemDetailPanel`

作用：

* 显示当前被点击物品的描述
* 若满足条件，支持显示残念补充内容

### 10. `BeadButton`

作用：

* 显示残缺念珠图标
* 根据当前物品是否可触发残念而高亮
* 点击后触发残念展示

### 11. `ClueCounter`

作用：

* 显示当前线索收集数，如 `1/10`

### 12. `DebateScreen`

作用：

* 辩论阶段容器
* 承载 NPC 发言、玩家操作区、物证选择、辩论历史、观点状态

### 13. `NpcSpeechPanel`

作用：

* 显示 NPC 当前发言

### 14. `DebateActionPanel`

作用：

* 显示追问输入框
* 显示出示物证入口
* 显示提交按钮

### 15. `EvidencePicker`

作用：

* 列出可用于辩论的物证

### 16. `ClaimStatusPanel`

作用：

* 显示当前主要观点列表及其状态

### 17. `ActEndScreen`

作用：

* 显示本幕结束文本
* 预留“进入下一幕”按钮或提示

### 18. `DebugPanel`

作用：

* 显示当前运行状态
* 显示线索、物证、观点、阶段等调试信息

# 7. 前端状态架构

本版前端状态分为三层：全局游戏状态、模块状态、纯 UI 状态。

## 7.1 全局游戏状态

这些状态必须进入统一 store。

### `phase`

当前大阶段。
可选值：

* `start`
* `opening`
* `exploration`
* `debate`
* `ending`

### `actId`

当前幕标识。
本版固定为 `act1_scene1`

### `scene`

当前场景对象。

### `zones`

当前场景的探索区域列表。

### `items`

当前场景全部物品列表。

### `selectedZoneId`

当前被选中的探索区域。

### `selectedItemId`

当前被查看的物品。

### `bead`

残缺念珠状态对象。

### `clues`

线索状态对象。

### `debate`

辩论状态对象。

### `history`

用于记录关键文本事件的历史摘要，便于本地调试。

---

## 7.2 模块状态

这些状态可以放在局部模块中，必要时同步到全局。

### `itemDetailExpanded`

物品详情面板是否展开。

### `beadHintVisible`

当前是否显示“可使用残缺念珠”的提示。

### `debateInputText`

当前辩论输入框内容。

### `selectedEvidenceId`

当前选中的物证。

### `debugPanelOpen`

调试面板是否展开。

---

## 7.3 纯 UI 状态

这些状态不进入业务状态层。

### `hoveredZoneId`

当前 hover 的区域。

### `hoveredItemId`

当前 hover 的物品。

### `loadingFlag`

模拟数据切换时的加载态。

### `toastMessage`

轻提示信息。

# 9. 内容提供层设计

本版不接 API，因此必须抽象出一个内容提供层，避免 UI 和未来接口强耦合。

## 9.1 统一内容提供接口

前端只认“内容提供器”，不关心其内部来源。

当前使用：

* `MockContentProvider`

未来替换为：

* `ApiContentProvider`

---

## 9.2 MockContentProvider 的职责

它负责提供以下数据：

* 开场演出文本
* 主界面初始场景数据
* 探索区域数据
* 物品数据
* 残缺念珠数据
* 辩论初始数据
* 追问响应模拟结果
* 出示物证响应模拟结果
* 幕终文本

---

## 9.3 Mock 数据约束

虽然是模拟数据，也必须满足真实设计约束：

* 一定包含残缺念珠
* 一定能收集到足够线索
* 一定能触发辩论
* 一定有 3~4 条主要观点
* 一定能用现有物证驳倒主要观点
* 一定能以“白姑去向”收束本幕

这和你的设计大纲保持一致。

# 10. 交互事件流

---

## 10.1 开始界面事件流

### 事件：点击开始游戏

输入：

* 用户点击“开始游戏”

结果：

* `phase` 从 `start` 变为 `opening`
* 加载开场演出数据
* 渲染 `OpeningScene`

---

## 10.2 开场演出事件流

### 事件：完成开场演出

输入：

* 用户点击“继续”或“跳过”

结果：

* 初始化第一幕场景数据
* 写入 `scene`、`zones`、`items`
* `phase` 变为 `exploration`
* 默认不显示残缺念珠图标
* 渲染 `MainGameScreen`

---

## 10.3 探索阶段事件流

### 事件：进入探索主界面

结果：

* 展示场景图、场景描述、区域列表、线索计数
* 默认选中第一个可探索区域或不选中区域
* 不自动展开任何物品详情

### 事件：点击区域

输入：

* `zoneId`

结果：

* 更新 `selectedZoneId`
* 渲染该区域下物品图标列表

### 事件：点击物品

输入：

* `itemId`

结果：

* 将该物品标记为 `isDiscovered = true`
* 将该物品标记为 `isExamined = true`
* 更新 `selectedItemId`
* 在 `ItemDetailPanel` 显示该物品普通描述
* 若该物品就是残缺念珠：

  * 设置 `bead.found = true`
  * 设置 `bead.visible = true`
  * 弹出获得重要道具提示
* 若该物品承载线索且首次获取：

  * 更新线索计数
* 若该物品 `beadReactive = true` 且 `bead.found = true`：

  * 设置 `bead.active = true`
  * 提示玩家可查看残念
* 若当前线索已达成辩论触发条件：

  * 允许触发辩论入口或直接进入辩论过场

### 事件：点击残缺念珠

前提：

* `bead.found = true`
* 当前物品 `beadReactive = true`

结果：

* 在当前物品描述下追加残念文本
* 将物品标记为 `isBeadUnlocked = true`
* 若残念对应新线索且首次获得：

  * 更新线索计数
* `bead.active` 恢复为 false，直到切换到下一个可感应物品

### 事件：线索达标

结果：

* 触发辩论过场
* `phase` 变为 `debate`
* 初始化 `debate` 状态

---

## 10.4 辩论阶段事件流

### 事件：进入辩论

结果：

* 展示 NPC 初始发言
* 展示主要观点列表
* 展示操作入口：追问 / 出示物证

### 事件：选择追问

输入：

* 玩家自由输入文本

结果：

* 调用 Mock 辩论响应逻辑
* 更新 NPC 发言
* 更新辩论历史
* 可能使当前观点更具体，但通常不直接驳倒观点

### 事件：选择出示物证

输入：

* `evidenceId`
* 可选附加文本

结果：

* 调用 Mock 辩论判断逻辑
* 判断该物证是否命中当前观点
* 若命中：

  * 当前观点状态更新为 `weakened` 或 `refuted`
  * `refutedCount + 1`
  * NPC 态度阶段后撤
* 若未命中：

  * NPC 进行防御性回应
* 写入辩论历史
* 判断是否切换下一个观点
* 判断是否满足辩论结束条件

### 事件：辩论结束

前提：

* 大部分主要观点已被驳倒
* `destinationUnlocked = true`

结果：

* `phase` 变为 `ending`
* 显示 NPC 吐露白姑去向的收束文本

---

## 10.5 幕终事件流

### 事件：本幕结束

结果：

* 展示结尾文本
* 预留“进入下一幕”按钮
* 当前版本点击后只回到开始页或停留在结束界面

# 11. 前端页面与布局要求

## 11.1 主界面布局要求

主界面至少包含以下区域：

* 顶部或左上：场景标题
* 中央上方：场景占位图
* 中央中部：场景描述
* 左侧或下方：探索区域列表
* 中部或右侧：当前区域物品图标列表
* 右侧或底部：物品详情面板
* 右下角：残缺念珠按钮
* 左下角：线索计数
* 可折叠角落：Debug Panel

## 11.2 辩论界面布局要求

辩论界面至少包含：

* NPC 当前发言区
* 当前主要观点状态区
* 玩家操作区
* 追问输入框
* 物证选择区
* 辩论历史区
* 结束态提示区

* 白姑去向文本

## 14.2 当前不能被写死的部分

UI 层不应直接假设：

* 物品数量固定
* 区域数量固定
* 观点数量固定
* NPC 语句长度固定
* 每条线索都一定来自同一类型物品

## 14.3 未来 API 兼容原则

现在的 Mock 数据字段名，就是未来 API 返回字段的第一版契约。
因此本版字段命名要稳定，不要一边写 UI 一边频繁改字段。

# 16. 验收标准

如果本版前端原型开发完成，应满足以下标准：

1. 本地打开即可游玩，不依赖后端
2. 玩家可以从开始界面一路玩到第一幕结束
3. 探索、残缺念珠、线索计数、辩论四个核心环节都可实际操作
4. 物证确实能在辩论中改变观点状态
5. 玩家不会在辩论中被卡死回探索
6. 全部内容均由 Mock 数据驱动，但前端结构已为未来 API 接入留出位置
7. Debug Panel 能帮助快速验证流程和状态正确性

# 17. 一句话总结

**这一版 Web 开发只做一个本地可玩的第一世第一幕单页原型：前端本地状态作为唯一权威，使用 Mock 数据和静态占位图跑通“开场演出 → 探索 → 获得残缺念珠 → 查看残念 → 收集线索 → 辩论驳倒 NPC → 幕终收束”的完整闭环，并为未来接入真实 API 保留稳定的数据契约。**
